How It Works

The Integration Pipeline

01
📂
Data Ingestion
Upload Excel or CSV exports from any museum collection management system. Objects, actors, events, and institutions are processed separately.
02
🗺️
CRM Mapping
Tabular columns are mapped to CIDOC CRM classes and properties using configurable field schemas. Museum-specific vocabularies are normalized.
03
🧠
Entity Resolution
LLM-powered semantic matching identifies identical actors, places, and objects across multiple institutional datasets with configurable similarity thresholds.
04
🕸️
Graph Storage
Resolved entities and their relationships are stored as nodes and edges in a Neo4j knowledge graph, ready for complex provenance queries.

Semantic Model

CIDOC CRM Mapping

Each spreadsheet entity type is mapped to internationally standardized CIDOC CRM classes, ensuring interoperability with other cultural heritage datasets worldwide.

Source Data CIDOC CRM Class Description
Museum objectE22 Human-Made ObjectPhysical artefacts in a collection, including all descriptive metadata
Person / DealerE39 Actor / E21 PersonIndividuals and groups involved in the creation, sale, or donation of objects
Acquisition eventE8 AcquisitionTransfer of ownership from one actor or institution to another
Production eventE12 ProductionThe creation of an object, including place and approximate time period
Location / ProvenanceE53 PlaceGeographic places of origin, storage, or transit along the provenance chain
Time-spanE52 Time-SpanDate ranges for events, normalized and linked across institutional calendars
InstitutionE74 GroupMuseums, auction houses, and other organizations participating in collection histories

Technical Architecture

Built for Research

Backend Stack
🐍
Django / Python
Web framework handling file uploads, session management, and orchestration of the integration pipeline
🕸️
Neo4j Graph Database
Native graph storage for nodes (entities) and relationships, queried via Cypher and the py2neo driver
📊
Pandas / OpenPyXL
Flexible Excel and CSV ingestion with schema detection, null handling, and column normalization
AI / NLP Layer
🧠
Large Language Models
Configurable LLM backend (Anthropic Claude, OpenAI) for semantic entity comparison and disambiguation
📐
Sentence Embeddings
Vector-based similarity search for candidate pair generation before LLM verification, reducing API costs
CIDOC CRM Ontology
📋
Configurable Field Mapping
Each institution's column names are mapped to CRM properties via a JSON configuration — no code changes required
🔄
Vocabulary Reconciliation
Free-text material types, techniques, and places are reconciled against authority vocabularies (AAT, TGN)
🌐
Linked Data Export
Graphs can be serialized to RDF/Turtle for publication as Linked Open Data, enabling federation with other LOD datasets
Deployment
🖥️
Self-Hosted
Runs on any institution's own servers — collection data never leaves the researcher's infrastructure
🔒
Session-Based Auth
Institution-level access control with configurable Neo4j and LLM credentials stored per session

Key Features

Designed for Museum Researchers

🏛️
Multi-Institution Support
Register multiple partner museums and integrate their data in a single federated graph, preserving institutional provenance for every record.
⚙️
Configurable Thresholds
Adjust entity resolution sensitivity per run. Higher thresholds prioritize precision; lower thresholds maximize recall across noisier datasets.
🔍
Integration Logging
Every integration step is logged in full — which records were matched, why, and what the LLM's reasoning was — for full audit trails.
📤
JSON Graph Export
Export the full knowledge graph as JSON for use in external visualization tools, D3.js networks, or downstream analysis pipelines.
🗺️
2D & 3D Visualization
Explore the integrated knowledge graph interactively in both 2D (D3.js force layout) and 3D (Three.js orbital) views directly in the browser.
💬
BecaChat Integration
Ask natural language questions about the knowledge graph — "Who acquired this object?" — answered by an LLM with direct access to the Neo4j graph.
